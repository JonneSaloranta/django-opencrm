# Contributing to django-opencrm

Thank you for your interest in contributing! 🎉
This project aims to provide a reusable Django CRM app for NGOs and small businesses.

We follow best practices to keep development easy, reproducible, and consistent.

---

## 🛠 Development Setup

### 1. Clone the repository

```bash
git clone https://github.com/yourusername/django-opencrm.git
cd django-opencrm
```

### 2. Create a virtual environment

```bash
python -m venv .venv
source .venv/bin/activate  # On Windows: .venv\Scripts\activate
```

### 3. Install dependencies

We use **pip-tools** to manage dependencies.

```bash
pip install pip-tools
pip-compile requirements.in
pip install -r requirements.txt
```

> `requirements.in` contains human-editable dev dependencies.
> `requirements.txt` is fully pinned and auto-generated by pip-compile.

---

## 📦 Dependency Management

- `pyproject.toml` → runtime dependencies (installed by users)
- `requirements.in` → direct dev dependencies
- `requirements.txt` → pinned dependencies (generated)

### Rules

- ❗ Do NOT edit `requirements.txt` manually
- ✔ Add new dependencies to:
  - `pyproject.toml` (runtime), or
  - `requirements.in` (dev only)

### Updating dependencies

```bash
pip-compile --upgrade
```

Or upgrade a specific package:

```bash
pip-compile --upgrade-package Django
```

## 🖤 Code Formatting

We use **Black** for code formatting.

Format all code:

```bash
black .
```

Check formatting without changing files:

```bash
black --check .
```

> Optional: consider using a pre-commit hook to enforce Black automatically.

---

## 🧪 Running Tests & Coverage

Run all tests:

```bash
pytest
```

Run tests with coverage reporting:

```bash
coverage run -m pytest
coverage report        # summary in terminal
coverage html          # generate HTML report in htmlcov/
```

> Note: `htmlcov/` is included in `.gitignore` and should not be committed.

---

## 🧹 Code Style

- Follow Python best practices (PEP 8)
- Keep code readable and simple
- Add docstrings where useful
- Prefer explicit over implicit

---

## 🚀 Submitting Changes

1. Fork the repository
2. Create a feature branch:

```bash
git checkout -b feature/my-feature
```

1. Make your changes
2. Run tests and ensure coverage passes
3. Commit with a clear message:

```bash
git commit -m "feat: add customer tagging support"
```

1. Push and open a Pull Request

---

## 🧾 Commit Message Guidelines

We follow a simplified **conventional commits** style:

- `feat:` → new feature
- `fix:` → bug fix
- `chore:` → maintenance / tooling
- `docs:` → documentation changes

Example:

```bash
feat: add support for exporting contacts to Excel
```
